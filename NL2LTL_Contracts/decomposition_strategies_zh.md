# 合约分解策略

给定系统需求（如"3个机器人搬运5个箱子"），如何将其分解为假设-保证（assume-guarantee）合约？

## 分解维度

### 静态分配（Static Allocation）
在规约阶段预先分配任务给机器人，无需调度器。
- Robot1 -> 箱子1, 3；Robot2 -> 箱子2
- 每个机器人有固定的任务序列
- 合约最简（变量最少），但缺乏灵活性

### 动态分配（Dynamic Allocation）
调度器（Dispatcher）观察机器人完成状态，动态分配任务。
- 任意机器人可以搬运任意箱子
- 调度器合约：`G(ri_done & !all_done -> F(ri_task))`
- 变量更多（需要调度器状态），但可容错、可适应

### 层级分解（Hierarchical Decomposition）
多层结构：高层规划器分解目标 -> 中层协调器分配子目标 -> 底层机器人执行。
- 适用于具有嵌套子任务的复杂任务
- 每层都有独立的假设-保证合约
- 比扁平组合更具可扩展性

### 空间分解（Spatial Decomposition）
按物理区域划分。Robot1 负责 A 区所有任务，Robot2 负责 B 区。
- 与静态分配类似，但按地理位置组织
- 适合仓库/工厂平面布局
- 减少机器人间的协调需求

### 功能分解（Functional Decomposition）
按能力划分。Robot1 负责搬运，Robot2 负责检测，Robot3 管理充电。
- 适用于异构机器人团队
- 每种机器人类型有不同的合约结构
- 合约反映能力差异

### 时序分解（Temporal Decomposition）
按阶段划分。阶段1：所有机器人取货；阶段2：所有机器人送货。
- 流水线/批处理模式
- 阶段之间需要同步合约
- 降低并发复杂度

### 混合（Hybrid）
以上方式的组合。例如：先按空间分区，每个区域内再动态分配。

## 策略选择

分解策略取决于系统的约束条件，而非任务本身：

| 约束条件 | 推荐策略 |
|---------|---------|
| 任务提前已知 | 静态分配 |
| 任务在线到达 | 动态分配 |
| 同构机器人（能力相同） | 动态分配（任意分配） |
| 异构机器人（能力不同） | 功能分解 |
| 明确的区域边界 | 空间分解 |
| 需要容错能力 | 动态分配（可重新分配） |
| 最小化合约复杂度 | 静态分配（无需调度器） |
| 嵌套子任务 | 层级分解 |
| 分阶段执行 | 时序分解 |

## Agent 辅助分解

让 LLM agent 辅助分解，分两步进行：

### 第一步：提取分解元信息

从自然语言需求中提取：

1. **智能体结构**：机器人数量和类型（同构/异构？）
2. **任务结构**：任务是否提前已知？在线到达？循环执行？
3. **共享资源**：是否有资源竞争？顺序约束？
4. **空间约束**：是否有区域划分？需要避碰？
5. **故障模式**：机器人故障时怎么办？

### 第二步：从模板库中选择

Agent 不需要从零发明分解方式，而是从一个**已验证的合约架构模板库**中选择。每个模板包含：

- 适用条件（何时使用此模式）
- 合约结构（哪些组件、接口定义）
- 变量归属（每个组件的输入/输出）
- 假设-保证关系

这遵循了 Iannopollo 等人提出的基于库的合约综合方法（DATE 2014）。

## 研究方向

核心研究问题：

> 给定同一个自然语言需求，能否自动生成多种合约分解方案，并比较哪种更优？

这需要完整的 pipeline：

```
自然语言需求
    |
    v
分解策略选择（选择哪种模式？）
    |
    v
合约生成（generate_contracts.py）
    |
    v
综合（slugs）+ 验证（nuxmv IC3）
    |
    v
评估（比较不同分解方案）
```

### 评估指标

| 指标 | 含义 |
|-----|------|
| 合约变量数 | 影响综合的可行性 |
| FSM 状态数 | 影响部署效率 |
| 容错能力 | 系统能否从机器人故障中恢复？ |
| 可扩展性 | 增加一个机器人需要修改多少？ |
| 验证时间 | IC3/BMC 精化检查的运行时间 |

`generate_contracts.py` 脚本处理中间步骤（合约生成、综合/验证流程）。待解决的开放问题：
- **第一步**：自然语言 -> 分解策略（需要领域知识 + 约束提取）
- **最后一步**：评估指标与自动化比较
